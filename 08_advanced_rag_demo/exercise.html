Simple RAG connects a retriever to a model. Advanced RAG measures and tunes the complete information path so better evidence reaches the reader model.

Learning objectives

  • Diagnose why character-based chunks can exceed an embedding model's token limit.
  • Create Markdown-aware chunks measured by the actual embedding tokenizer.
  • Remove duplicate chunks and preserve source/start metadata.
  • Use normalized embeddings with cosine distance in FAISS.
  • Visualize semantic neighborhoods with PaCMAP.
  • Use a model's chat template instead of manually guessing prompt tokens.
  • Retrieve many candidates, rerank them with ColBERT, and keep only the best context.
  • Compare reranking configurations and record evidence-based evaluation results.

4. Try Markdown-aware character chunking

This first attempt is intentionally diagnostic. It respects Markdown structure but measures size in characters.

Colab cell - Character chunks
from langchain_text_splitters import RecursiveCharacterTextSplitter

MARKDOWN_SEPARATORS = [
    "\n#{1,6} ", "```\n", "\n\\*\\*\\*+\n", "\n---+\n",
    "\n___+\n", "\n\n", "\n", " ", "",
]

character_splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,
    chunk_overlap=100,
    add_start_index=True,
    strip_whitespace=True,
    separators=MARKDOWN_SEPARATORS,
)

character_chunks = []
for doc in raw_knowledge_base:
    character_chunks.extend(character_splitter.split_documents([doc]))

The separator hierarchy first tries headings and code fences, then weaker boundaries. start_index makes it possible to trace a chunk back into its source.

Important diagnosis: 1,000 characters is not 1,000 tokens. Chunks above the model's 512-token limit are silently truncated during embedding, which can discard useful text and weaken retrieval.

6. Replace it with tokenizer-aware chunking

Colab cell - Token-aware splitter
def split_documents(chunk_size, knowledge_base, tokenizer_name=EMBEDDING_MODEL_NAME):
    tokenizer = AutoTokenizer.from_pretrained(tokenizer_name)
    splitter = RecursiveCharacterTextSplitter.from_huggingface_tokenizer(
        tokenizer,
        chunk_size=chunk_size,
        chunk_overlap=int(chunk_size / 10),
        add_start_index=True,
        strip_whitespace=True,
        separators=MARKDOWN_SEPARATORS,
    )

    chunks = []
    for doc in knowledge_base:
        chunks.extend(splitter.split_documents([doc]))

    unique_chunks = []
    seen_text = set()
    for doc in chunks:
        if doc.page_content not in seen_text:
            seen_text.add(doc.page_content)
            unique_chunks.append(doc)

    return unique_chunks

docs_processed = split_documents(512, raw_knowledge_base)
  • Chunk size now uses the embedding tokenizer.
  • Overlap remains 10% to protect boundary context.
  • Exact duplicate text is removed so repeated pages do not dominate retrieval.
  • Run the second histogram and verify the distribution is aligned with the model limit.

7. Build cosine FAISS retrieval

Colab cell - Vector database
from langchain_huggingface import HuggingFaceEmbeddings
from langchain_community.vectorstores import FAISS
from langchain_community.vectorstores.utils import DistanceStrategy

embedding_model = HuggingFaceEmbeddings(
    model_name=EMBEDDING_MODEL_NAME,
    model_kwargs={"device": "cuda"},
    encode_kwargs={"normalize_embeddings": True},
)

knowledge_vector_database = FAISS.from_documents(
    docs_processed,
    embedding_model,
    distance_strategy=DistanceStrategy.COSINE,
)

Cosine similarity compares vector direction. Normalizing the vectors and configuring the index consistently prevents a metric mismatch.

13. Assemble the advanced RAG function

The notebook's answer_with_rag function performs these steps:

  1. Retrieve num_retrieved_docs candidates from FAISS.
  2. Keep content together with source metadata.
  3. Optionally rerank candidate text with ColBERT.
  4. Keep num_docs_final documents.
  5. Number and concatenate documents into context.
  6. Fill the model-specific prompt template.
  7. Generate the answer and return both answer and source records.
Source preservation: the educational implementation maps reranked result indices back to original metadata. Do not throw metadata away before reranking if you need citations or traceability.

Advanced concepts retained from the cookbook

  • Lost in the middle: excessive context can reduce a model's use of relevant evidence.
  • Token-limit alignment: chunking must respect the embedding model, not only character count.
  • Metric alignment: normalized vectors and cosine distance must be configured consistently.
  • Two-stage retrieval: fast broad recall followed by slower precise reranking.
  • Reader context capacity: retrieved documents plus instructions and question must fit the reader window.
  • Iterative evaluation: tune one subsystem at a time against a stable question set.
